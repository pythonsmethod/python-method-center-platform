1. Краткий вердикт

Проект не готов встретить первого клиента и далёк от этого. Главный разрыв — между документацией и продуктом: написано ~7 300 строк проектных документов (фактически готовые спецификации всех шести пунктов цели), но в коде реализовано примерно полтора пункта из шести.

Построен аккуратный технический фундамент (Next.js, схема БД, аутентификация, загрузка документов, журнал аудита), однако пять из шести запускаемых поверхностей либо отсутствуют, либо являются заглушками: нет сайта (есть только список ссылок-заглушек), нет приёма оплат (нет даже зависимости Stripe), нет ни одного AI (нет AI-SDK в зависимостях), кабинет Карена сведён к read-only списку документов, рабочего места поддержки нет. Вдобавок продукт нигде не развёрнут и БД не подключена (намеренно), а оферта/политика возврата существуют только как PDF без юридической проверки перед первой оплатой.

Топ-3 блокера запуска

  1. Нет публичного сайта и нет приёма оплат. Человеку буквально некуда прийти и нечем заплатить. Это не «доделать», это «построить».
  2. Нет ни одного AI и нет поверхности оценки Карена. Пункты 3/5/6 («анализ анализов», AI-ассистент Карена, AI для клиентов) не существуют в коде; нет таблиц messages, karen_reviews, assessment, ai_session, red_flag_event.
  3. Инфраструктура и юридическая база не введены в строй. Supabase не подключён, деплоя нет, оферта/возврат/приватность не прошли юридическую проверку — а без неё нельзя брать первую оплату.

Вторая по важности проблема — дрейф цели: основной объём труда вложен в философию и методологию (Конституция из 11 разделов, «Видение на 20 лет», «Принципы масштабирования», Конституция центра знаний), то есть в задачи «послезавтра», тогда как пять из шести поверхностей «сегодня» не построены.

4. Дрейф цели

Проект ушёл от запуска вниз — в глубину смысла, вместо того чтобы уйти вперёд — к рабочему продукту.

  1. Философия вместо поверхностей. «Утверждены» разделы, не приближающие клиента: constitution_10_vision_20_years, constitution_09_scaling_principles, constitution_of_knowledge_center, «основы методологии». По эталонной цели — V2+. Параллельно сайт, оплаты и оба AI не существуют.
  2. Решение завтрашних задач. Масштабирование, центр знаний, обучение AI на кейсах, версионирование базы знаний — про рост после первых клиентов.
  3. Бесконечное углубление документации. Сигнатура дрейфа — ~7 300 строк документов к ~3 400 строкам кода, где код покрывает ~1,5 из 6 целей. Появились мета-документы об документах (DOCUMENT_INVENTORY, CODEX_AUDIT, DATA_MODEL_OPEN_DECISIONS, DATA_MODEL_TO_SUPABASE_MAPPING, CANONICAL_LIFECYCLE_STATUS_MODEL) — согласования спецификаций, ещё не превращённых в продукт.
  4. Незакрытые противоречия из собственного аудита (CODEX):
    • Конфликт имени бренда — НЕ ЗАКРЫТ. Заголовки Конституции до сих пор называют центр «РЕАБИЛИТАЦИЯ БЕЗ ГРАНИЦ», при том что бренд-идентичность фиксирует: имя = Python Method, а «Реабилитация без границ» — только слоган. Прямое брендово-юридическое противоречие перед публикацией сайта и оферты.
    • Тройной дрейф статус-энумов. case_status различается в трёх местах: миграция (created/awaiting_onboarding/ready_for_review/in_review/active_support/…) ≠ DATA_MODEL_V1 (account_created/…/analysis_complete/…) ≠ SUPABASE_SCHEMA_V1 (intake/under_review/active/paused/closed). Код уже разошёлся со спеками.
Вывод: глубина метода — не недостаток сама по себе, но как приоритет сейчас она и есть отклонение. Всё из п.1–3 нужно заморозить как V2 и не тратить на это внимание до первых клиентов.

6. План устранения (приоритизированный)

Уровень A — блокеры, требующие решения владельца

#Что делаемКто решает
A1Подключить Supabase, применить миграции, проверить вход+онбординг+загрузку на живой БДВладелец → разработка
A2Завести Stripe; выбрать валюту и цены тарифов 5/15 недельВладелец
A3Юр-проверка оферты, возврата, приватности; тексты дисклеймеров и чекбоксовВладелец + юрист
A4Выбрать AI-провайдера и бюджет (пп. 3/5/6)Владелец
A5Подтвердить сокращение объёма MVP (заморозить V2-философию)Владелец
A6Закрыть конфликт имени бренда: заголовки Конституции → «Python Method»Владелец → разработка

Уровень B — критично для запуска (стройка после A)

  1. Сайт (п.1): 8 контент-страниц по WEB_ARCHITECTURE без обещаний результата + CTA → регистрация. (можно начинать сразу.)
  2. Оплаты (п.2): Stripe Checkout + вебхуки с идемпотентностью + активация service_period по оплате + дисклеймер/чекбоксы + аудит.
  3. Анализ анализов (п.3): сущность assessment; AI-структурирование документов; поверхность, где Карен формулирует и публикует оценку.
  4. Кабинет Карена (п.4): messages + karen_reviews; карточка кейса; очереди; написание/публикация заключений; статусы (только Карен).
  5. AI Карена (п.5) и AI для клиентов (п.6) в рамках AI_GUARDRAILS; red_flag_event + двойная маршрутизация (физ./мед. → Карен, психол./кризис → Anna/Support).
  6. Обязательные политики в рантайме: гейт 21+ и Care Recipient; ретеншн/удаление; field-level доступ.
  7. Согласование статус-энумов кода с CANONICAL_LIFECYCLE_STATUS_MODEL.
  8. Деплой на Vercel + домен + Supabase redirect URLs.

Уровень C — желательное (не блокирует первого клиента)

  • Сквозной smoke-тест по «10 MVP Launch Checklist» (18 шагов).
  • Рабочее место поддержки (тикеты) — при MVP роль Anna частично вручную.
  • PDF-выгрузка заключений, уведомления, расширенная аналитика — V2.

7. Что откладываем в V2

  • Конституция р.10 «Видение на 20 лет», р.09 «Масштабирование», Конституция центра знаний, «основы методологии» как предмет доработки.
  • База знаний, обучение AI на кейсах, версионирование/публикация знаний, извлечение паттернов.
  • Будущие продукты (Python Elixir, Python Peptide), мультиязычный контент сверх принципа, полноценная CMS, брендинг-дизайн сверх минимума.
  • Мобильное приложение, голосовой режим Карена, внутренние заметки, кнопка срочного звонка, автоприоритизация, командные роли, академия, семейные аккаунты, скидки, авто-PDF-отчёты.
  • Авто-переоценка, авто-биллинг/подписки, рассрочки, авто-возвраты, расширенная аналитика.
Итог. Технический фундамент — крепкий и честный (auth, RLS, аудит, приватный Storage). Но по эталонной цели проект на стадии «фундамент + один угол дома»: из шести поверхностей реально стоит полторы. Ближайший разумный шаг — решения владельца уровня A, после чего строить B1→B8 строго по готовым спекам, не углубляя документацию дальше.
