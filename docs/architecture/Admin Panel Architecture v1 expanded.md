# Admin_Panel_Architecture_v1

## Статус документа

Документ №4 Тома 2 «Архитектура запуска».

Основан на:

- Launch Architecture Foundation v1
- Support System Architecture v1
- Client Cabinet Architecture v1

-----

# 1. Назначение документа

Этот документ определяет архитектуру административной панели центра.

Административная панель является рабочей операционной средой Карена и команды центра.

Её задача — не просто хранить клиентов, документы и сообщения, а организовать работу так, чтобы Карен мог сопровождать большее количество людей без хаоса, перегрузки и потери качества.

-----

# 2. Главный принцип панели

## Система работает на Карена, а не Карен работает на систему

Карен не должен вручную искать:

- документы;
- историю клиента;
- старые сообщения;
- прошлые рекомендации;
- новые анализы;
- текущие обращения;
- сроки сопровождения;
- приоритетные ситуации.

Система должна сама показывать Карену то, что требует внимания.

-----

# 3. Цель админ-панели

Админ-панель создаётся для того, чтобы:

- уменьшить хаос;
- убрать работу через мессенджеры;
- сохранить всю историю клиента в одном кейсе;
- защитить время Карена;
- ускорить обработку обращений;
- помочь Карену видеть приоритеты;
- дать AI возможность помогать Карену;
- создать масштабируемую систему сопровождения.

-----

# 4. Кто использует админ-панель

## Карен

Главный пользователь панели.

Карен работает с:

- новыми кейсами;
- активными сопровождениями;
- обращениями клиентов;
- приоритетными ситуациями;
- рекомендациями;
- заключениями;
- AI-Ассистентом.

## Команда центра

В будущем команда может работать с:

- оплатами;
- загрузками документов;
- организационными вопросами;
- технической поддержкой;
- подготовкой кейсов;
- административными задачами.

На этапе MVP главной ролью остаётся роль Карена.

-----

# 5. Главная панель дня

После входа Карен должен видеть не всю базу клиентов, а рабочую картину дня.

## Панель дня показывает:

- сколько новых кейсов готово к рассмотрению;
- сколько новых обращений поступило;
- сколько приоритетных обращений ожидает внимания;
- сколько документов и анализов загружено;
- какие сопровождения завершаются;
- какие клиенты ожидают ответа Карена;
- какие кейсы ожидают ответа клиента;
- какие задачи требуют действия сегодня.

-----

# 6. Утренний обзор через AI-Ассистента

Карен может начать день с команды AI-Ассистенту:

> Что у нас сегодня?

AI-Ассистент отвечает кратко:

- новых кейсов: 5;
- обычных обращений: 18;
- приоритетных обращений: 3;
- новых анализов: 7;
- сопровождений завершается в ближайшие 7 дней: 4;
- клиентов ждут ответа Карена: 12;
- клиентов ждут ответа от себя: 6.

Цель утреннего обзора — дать Карену понимание нагрузки без ручного просмотра всей системы.

-----

# 7. Рабочие очереди

Карен работает не с хаотичным списком клиентов, а с очередями.

## 7.1 Новые кейсы

Сюда попадают клиенты, которые:

- оплатили услугу;
- прошли онбординг;
- загрузили материалы;
- завершили формирование кейса;
- готовы к первичному рассмотрению Кареном.

Для каждого нового кейса отображается:

- имя клиента;
- продукт;
- дата оплаты;
- дата завершения онбординга;
- количество загруженных документов;
- маршрут клиента;
- краткая AI-сводка.

-----

## 7.2 Активные сопровождения

Список клиентов, у которых сейчас активен период сопровождения.

Отображается:

- имя клиента;
- тип сопровождения;
- дата начала;
- дата окончания;
- последний контакт;
- текущий статус;
- наличие открытых обращений.

-----

## 7.3 Обычные обращения

Стандартная очередь вопросов клиентов.

Примеры:

- вопрос по рекомендации;
- уточнение по документам;
- сообщение о плановом изменении;
- просьба пояснить ответ;
- новая информация без признаков срочности.

-----

## 7.4 Приоритетные обращения

Сюда попадают обращения, связанные с существенными изменениями.

Примеры:

- ухудшение состояния;
- госпитализация;
- назначение операции;
- новые серьёзные симптомы;
- резкое изменение самочувствия;
- важные назначения врача.

Приоритетные обращения отображаются отдельно от обычной очереди.

-----

## 7.5 Критические обращения

Критический статус назначается только Кареном.

Клиент не может самостоятельно поставить обращению критический статус.

Это защищает систему от злоупотреблений кнопкой срочности.

-----

## 7.6 Ожидают ответа клиента

Сюда попадают кейсы, где Карен уже задал уточняющий вопрос, но клиент ещё не ответил.

Такие обращения не должны оставаться в основной очереди Карена.

-----

## 7.7 Завершающиеся сопровождения

Отображает клиентов, у которых сопровождение скоро заканчивается.

Например:

- через 14 дней;
- через 7 дней;
- через 3 дня;
- сегодня.

Цель — заранее подготовить завершение периода сопровождения и предложить продление.

-----

# 8. Карточка кейса

Карточка кейса — главный рабочий экран Карена.

## Главный принцип карточки:

Сначала текущее событие.

Потом история.

Карен сначала должен видеть, что клиент написал сейчас, а уже затем при необходимости обращаться к истории кейса или AI-сводке.

-----

# 9. Верхний блок карточки

В верхней части карточки отображается:

- имя клиента;
- статус сопровождения;
- активный продукт;
- дата начала сопровождения;
- дата окончания сопровождения;
- маршрут клиента;
- уровень приоритета;
- последнее обращение клиента.

-----

# 10. Последнее обращение клиента

Последнее обращение клиента отображается первым.

Оно содержит:

- текст обращения;
- дату и время;
- тип обращения;
- прикреплённые файлы;
- статус обращения;
- уровень приоритета.

Это позволяет Карену сразу понять, что произошло сейчас.

-----

# 11. История кейса

В карточке кейса должна быть доступна полная история взаимодействия человека с центром.

История включает:

- дату первого обращения;
- онбординг;
- цели клиента;
- документы;
- анализы;
- обращения;
- ответы Карена;
- рекомендации;
- периоды сопровождения;
- завершённые этапы;
- возобновления сопровождения.

История не удаляется после завершения сопровождения.

-----

# 12. Документы и анализы

В карточке кейса отдельный раздел содержит:

- загруженные анализы;
- медицинские документы;
- выписки;
- обследования;
- изображения;
- дополнительные файлы.

Документы должны быть связаны с датой загрузки и конкретным обращением, если они были прикреплены к обращению.

-----

# 13. Рекомендации и заключения

Отдельный раздел содержит:

- первичное заключение Карена;
- рекомендации;
- корректировки;
- последующие ответы;
- материалы, отправленные клиенту;
- инструкции из библиотеки центра.

Эти материалы должны сохраняться в кейсе и быть доступны Карену при повторных периодах сопровождения.

-----

# 14. Периоды сопровождения

В карточке отображаются все периоды сопровождения клиента.

Например:

## Период 1

Сопровождение 5 недель

Дата начала: 01.07.2026

Дата окончания: 05.08.2026

## Период 2

Сопровождение 15 недель

Дата начала: 01.11.2026

Дата окончания: 14.02.2027

Кейс остаётся единым, даже если сопровождение включалось и выключалось много раз.

-----

# 15. AI-Ассистент Карена

AI-Ассистент является отдельной сущностью внутри админ-панели.

Его задача — усиливать эффективность Карена.

AI-Ассистент не заменяет Карена и не принимает решения вместо него.

-----

# 16. Основной принцип AI-Ассистента

## AI усиливает Карена

AI нужен для того, чтобы:

- быстро искать информацию;
- сокращать ручную работу;
- готовить черновики ответов;
- находить прошлые рекомендации;
- переводить материалы;
- структурировать хаос;
- помогать Карену работать с большим количеством людей.

-----

# 17. Режим 1: Информационный

AI отвечает на вопросы Карена по кейсу.

Примеры команд:

> Расскажи коротко историю клиента.

> Что изменилось с прошлого контакта?

> Какие анализы были загружены за последние 3 месяца?

> Когда клиент сообщил об операции?

> Что я рекомендовал в прошлый раз?

AI даёт Карену краткую сводку, не заставляя его перечитывать всю историю.

-----

# 18. Режим 2: Поисковый

AI ищет конкретные данные в истории кейса.

Примеры:

> Какие были тромбоциты за последние месяцы?

> Найди последний анализ крови.

> Покажи документы, связанные с операцией.

> Когда клиент последний раз писал о температуре?

AI не выдумывает данные. Если информации нет в кейсе, AI сообщает, что данных нет.

-----

# 19. Режим 3: Подготовительный

AI помогает подготовить ответ клиенту.

Карен может сказать:

> Подготовь ответ по этому обращению.

> Объясни клиенту простыми словами.

> Составь ответ от моего имени.

> Переведи документ и подготовь краткое объяснение.

AI формирует черновик.

Черновик не отправляется автоматически.

-----

# 20. Режим 4: Исполнительный

AI может выполнить действие только после команды или подтверждения Карена.

Примеры:

- отправить одобренный ответ;
- прикрепить ссылку;
- прикрепить стандартную рекомендацию;
- отправить инструкцию;
- сохранить ответ в кейсе;
- пометить обращение как завершённое.

-----

# 21. Связка Карен → AI → Клиент

Рабочая модель:

1. Клиент создаёт обращение.
1. Карен открывает обращение.
1. Карен даёт AI команду.
1. AI готовит черновик или действие.
1. Карен проверяет.
1. Карен одобряет.
1. AI отправляет клиенту.
1. Ответ сохраняется в истории кейса.

-----

# 22. Пример работы AI-Ассистента

Клиент пишет:

> У меня вздутие. Что делать?

Карен говорит AI:

> Подготовь ответ по нашей рекомендации от вздутия.

AI находит утверждённый материал из библиотеки центра.

AI готовит черновик.

Карен проверяет и говорит:

> Одобряю. Отправить.

AI отправляет клиенту и сохраняет ответ в кейсе.

-----

# 23. Библиотека материалов центра

AI-Ассистент должен использовать утверждённую библиотеку материалов.

Библиотека может содержать:

- стандартные рекомендации;
- инструкции;
- ссылки;
- шаблоны сообщений;
- организационные ответы;
- часто используемые материалы Карена.

-----

# 24. Ограничение библиотеки

AI не должен самостоятельно создавать методологические рекомендации как новые правила центра.

AI использует только:

- материалы, утверждённые Кареном;
- команды Карена;
- данные из конкретного кейса;
- тексты, которые Карен одобрил.

-----


# 24a. Управление знаниями: роли Карена и Админа

Канонические роли в управлении знаниями (Knowledge Entry):

**Карен — владелец методологии и одобрения знаний.** Карен может:

- создавать знания;
- изменять знания;
- одобрять знания;
- отзывать знания;
- одобрять изменения методологии и клинической логики.

**Админ — техническое управление, публикация и доступ.** Админ может:

- публиковать;
- снимать с публикации;
- архивировать;
- версионировать;
- управлять доступом;
- управлять видимостью.

**Админ НЕ может:**

- одобрять методологию;
- одобрять клиническую логику;
- одобрять содержание знаний.

**Жизненный цикл знания (Knowledge lifecycle):**

Draft → Karen Review → Karen Approved → Published → Archived.

Поля владения (будущая схема, без SQL): `created_by`, `reviewed_by`, `approved_by_karen`, `published_by_admin`.

Каждое одобрение, отзыв, публикация и архивация знания должны фиксироваться в аудите.

-----
# 25. Голосовой режим

Голосовой режим является важной функцией панели Карена.

Карен может:

- спросить AI голосом;
- получить краткий голосовой ответ;
- продиктовать ответ клиенту;
- попросить AI оформить текст;
- одобрить отправку голосовой командой.

Это особенно важно для ускорения работы и снижения нагрузки на Карена.

-----

# 26. Статусы обращений

Обращения могут иметь статусы:

- новое;
- ожидает рассмотрения;
- в работе;
- ожидает ответа клиента;
- ответ подготовлен;
- отправлено клиенту;
- завершено.

-----

# 27. Статусы кейсов в админ-панели

Кейс может иметь статусы:

- аккаунт создан;
- ожидает онбординг;
- кейс сформирован;
- готов к рассмотрению;
- анализ завершён;
- сопровождение активно;
- сопровождение не активно;
- сопровождение возобновлено;
- сопровождение завершено.

-----

# 28. Уведомления Карена

Карен получает уведомления о:

- новых кейсах;
- новых обращениях;
- приоритетных обращениях;
- новых анализах;
- ответах клиента;
- завершении сопровождения;
- необходимости действия.

-----

# 29. Что не должно быть в панели MVP

В MVP не нужно добавлять:

- сложную аналитику;
- социальную ленту;
- лишние графики;
- внутренние чаты команды;
- функции, не связанные с запуском;
- полную базу знаний кейсов;
- обучение AI на всех кейсах.

Это можно перенести в Version 2.

-----

# 30. MVP Admin Panel v1

Для запуска обязательно:

- главная панель дня;
- рабочие очереди;
- карточка кейса;
- последнее обращение клиента;
- документы и анализы;
- рекомендации;
- история кейса;
- AI-Ассистент Карена;
- подготовка ответов;
- библиотека материалов;
- статусы обращений;
- статусы кейсов.

-----

# 31. Version 2

После запуска можно развивать:

- расширенную аналитику;
- полноценную базу знаний;
- обучение AI через реальные кейсы;
- статистику эффективности;
- командные роли;
- расширенные отчёты;
- автоматическую приоритизацию;
- продвинутый голосовой режим.

-----

# 32. Итоговый принцип

Административная панель должна превращать хаос обращений, документов, анализов и сообщений в понятную рабочую систему.

Карен должен видеть главное, быстро получать контекст, работать через AI-Ассистента и сохранять качество сопровождения даже при росте количества клиентов.
