## 1. Главная страница — BLOCKED

`app/(public)/page.tsx` — это техническая dev-страница: заголовок «P0-001 scaffold», «Runnable web platform foundation», сетка ссылок на маршруты. Ни слова о продукте, тарифах, Карен, оферте. Всё на английском. Реального клиента на неё отправлять нельзя. Технически страница работает, но как витрина для первого клиента — блокер (либо заменить контент, либо вести клиента прямой ссылкой на /login и считать главную отложенной).

## 2. Регистрация / вход — READY

`/login` + `lib/auth/actions.ts`: Supabase Auth email+password, sign-up и sign-in, redirect с защитой от open redirect (`sanitizeNextPath`), logout, `/auth/callback` для e-mail-подтверждения. Все защищённые маршруты корректно редиректят на `/login?next=...`. Код готов. Условие: Supabase-проект должен быть развернут и env-переменные заданы (см. блокер деплоя ниже).

## 3. Клиентский кабинет — PARTIAL

`/cabinet` показывает: сессию, статус кейса (status/urgency/direction/title), панель загрузки документов со списком загруженного. Работает. Отсутствуют: сообщения, оплата/биллинг, история кейса, какие-либо следующие шаги для клиента. В текстах интерфейса — служебные формулировки («Cabinet functionality is still reserved for future implementation»). Для минимального сценария «зарегистрировался → заполнил анкету → загрузил документы» — достаточно.

## 4. Создание кейса — READY

`lib/onboarding/actions.ts`: при сабмите онбординга создаётся `client_cases` (status `ready_for_review`, title = цель, summary = описание ситуации), один кейс на профиль (unique constraint), запись в `onboarding_submissions`, `consent_records`, `audit_logs`. Идемпотентно при повторном сабмите (кейс не дублируется). Работает.

## 5. Онбординг — PARTIAL

Форма: ФИО, телефон, для кого запрос (self/family), цель, описание ситуации, чекбокс согласия. Цепочка записей полная и аудируется. Частично, потому что: (а) согласие — одна общая фраза «I consent to storing this onboarding information», без ссылки на оферту/политику и без фиксации `offer_acceptance` (записывается только `data_processing`); (б) анкета сильно уже архитектурного интейка (нет направления, срочности, медицинского контекста); (в) интерфейс на английском. Функционально путь проходим.

## 6. Загрузка документов — READY

Лучший блок платформы. Клиент грузит в приватный бакет `client-documents` по пути `{user_id}/{case_id}/{document_id}/{filename}`, сервер валидирует тип/размер/путь, проверяет фактическое наличие объекта в storage, пишет метаданные и аудит. Storage RLS привязан к `auth.uid()` в первом сегменте пути. Условие: бакет и storage-политики создаются вручную через Dashboard (`supabase/storage_manual_setup.md`) — это шаг деплоя, не кода.

## 7. AI-сводка кейса — MISSING

AI-кода нет вообще: папка `ai/` содержит только README «no AI runtime yet». Нет ни сводки кейса, ни клиентского AI, ни AI-ассистента Карен. Для первого клиента сводку заменяет ручное чтение анкеты и документов Карен.

## 8. Кабинет Карен / админ-панель — PARTIAL

`/admin` защищён ролью (support/admin из `profiles.role`, роль захардена триггером — клиент не может назначить её себе; это сделано хорошо). Но функционально есть только одно: read-only список загруженных документов (`/admin/documents`) с открытием файла по короткоживущей signed URL (60 сек, service role).

Критичный пробел: **Карен не видит ни кейсы, ни анкеты онбординга** — ни списка клиентов, ни текста «situation description», ни статуса кейса. Нет управления статусом, нет заметок, нет очередей. Обходной путь на первого клиента — читать таблицы напрямую в Supabase Dashboard, но это работа вне платформы.

## 9. Сообщения клиент ↔ команда — MISSING

Никакого UI сообщений нет ни у клиента, ни у команды. В схеме есть `support_requests` (с RLS), но ни одна страница их не создаёт и не читает. `/support` — заглушка «No runtime flow yet». Коммуникация с первым клиентом возможна только вне платформы (email/мессенджер).

## 10. Оплата тарифа — MISSING

`/payment` — заглушка «Provider not connected». Stripe не подключён, checkout нет, webhook нет, тарифы (5/15 недель) нигде не представлены. Таблица `payments` в схеме есть. Дополнительно: временная RLS-политика `payments_insert_own_placeholder` позволяет аутентифицированному клиенту самому вставить запись об оплате (в т.ч. со статусом `paid`) — пока оплат нет, это не эксплуатируемо, но политику нужно убрать до включения любой платёжной логики.

## 11. Юридические документы — BLOCKED

Оферта («Оферта новая версия 2.pdf») и Протокол безопасности лежат только в репозитории (`docs/legal/`, `docs/safety/`) — на сайте они не опубликованы. Нет страниц оферты/политики конфиденциальности, чекбокс онбординга на них не ссылается, `offer_acceptance` не фиксируется. Принимать деньги и медицинские документы от реального клиента без опубликованной оферты и явного её принятия нельзя — это блокер.

## 12. Red flags / безопасность — PARTIAL

Безопасность платформы (позитив): RLS на всех таблицах по принципу «клиент видит только своё»; роль и статус профиля защищены security-definer-триггером; клиент не может менять статусы документов и переносить их между кейсами (триггер P0-007A); бакет приватный; service role только на сервере; аудит-лог пишется. Замечания: (а) клиент через прямой insert в `uploaded_documents` может задать произвольные `metadata.storage_bucket` / `storage_path` — staff-роут `/view` доверяет им при генерации signed URL (риск для сотрудника открыть чужой объект, эксфильтрации клиенту нет); (б) нет `middleware.ts` с обновлением Supabase-сессии — длинные сессии могут протухать; (в) placeholder-политики insert для `payments` и `escalation_events` надо закрыть.

Red-flag workflow (негатив): по MVP Build Plan он «MVP-required, not deferred», но в рантайме отсутствует полностью — есть только таблица `escalation_events` и enum'ы. Нет детекции, нет маршрутизации «физическое → Карен / психологическое → Анна», нет очередей, нет даже статического экрана с экстренными контактами. Для первого клиента обязателен хотя бы ручной протокол.

## 13. История кейса — MISSING

Таблица `case_lifecycle_events` создана, но в неё никто не пишет (онбординг пишет только в `audit_logs`), и никакого UI истории нет ни у клиента, ни у Карен. Для первого клиента историю заменяет аудит-лог (читаемый через Supabase Dashboard).

## Кросс-блочный блокер: платформа не развернута

Прод-URL, на который можно отправить клиента, на сегодня не подтверждён.

1. **Нет развернутого прода**: Vercel + Supabase (миграции, бакет, storage-политики, redirect URLs, staff-профиль для Карен) — всё в статусе «manual, pending».

5. **Карен не видит кейс и анкету в админке** — только документы; работать с кейсом через платформу она не может.
6. **Главная страница — dev-заглушка**, клиента на неё отправлять нельзя.
7. **Red-flag протокол в рантайме отсутствует**, хотя объявлен обязательным для MVP.

## 2) Закрыть за 24 часа

- Развернуть прод: Vercel-проект, env (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`), применить 5 миграций, создать бакет `client-documents` и storage-политики по `storage_manual_setup.md`, настроить Auth redirect URLs, создать staff-профиль Карен. Прогнать смоук-тест из deployment.md (16 шагов).
- Заменить главную страницу и служебные тексты кабинета/онбординга на минимальный клиентский контент (можно одну простую страницу «что делать дальше»).
- Опубликовать оферту и политику обработки данных как страницы/PDF на сайте; чекбокс онбординга — со ссылкой на них; писать в `consent_records` дополнительно `offer_acceptance` с версией документа.

## 3) Закрыть за 72 часа

- **Оплата**: либо Stripe Payment Link / Checkout на два продукта с ручной фиксацией в `payments` (полноценный webhook-флоу за 72 ч рискован), либо осознанное решение «первый клиент платит по счёту вне платформы» с записью в `payments` вручную. До этого — удалить RLS-политику `payments_insert_own_placeholder`.
- **Админ-вид кейса для Карен**: список кейсов + просмотр анкеты онбординга (payload `onboarding_submissions`) и статуса кейса. Без этого Карен работает через Supabase Dashboard — допустимо только как временный костыль.
- **Канал связи**: минимум — UI поверх существующей таблицы `support_requests` (создание запроса клиентом + просмотр staff'ом), либо официально зафиксированный внешний канал (email), указанный в кабинете.
- **Red flags, ручная версия**: статичный блок с экстренными контактами в кабинете и онбординге + внутренний регламент для Карен/Анны по «Протоколу безопасности» (PDF уже есть).
- Закрыть мелкие дыры: политика insert `escalation_events`, доверие к `metadata.storage_bucket` в staff-роуте `/view`, middleware для обновления сессии.

## 4) Можно отложить после запуска

- AI-сводка кейса и любой AI-рантайм (Карен читает документы вручную).
- История кейса (UI + запись `case_lifecycle_events`) — аудит-лог уже пишется.
- Автоматический red-flag workflow (детекция, очереди, маршрутизация) — при наличии ручного протокола.
- Полный Stripe-флоу с webhook и `service_periods`.
- Support-workspace, admin-заметки, направление/срочность в онбординге, локализация, refund-автоматизация.

## 5) Финальный вердикт

**Сегодня принимать первого клиента через платформу нельзя.** Прод не развернут, оплату принять невозможно, оферта не опубликована, у Карен нет рабочего вида кейса, у клиента нет канала связи.

**За 7 дней возможен ограниченный (гибридный) запуск**, и это реалистично: ядро пути «регистрация → онбординг → создание кейса → загрузка документов → просмотр документов staff'ом» написано, собирается и сделано аккуратно (RLS, аудит, приватный storage). Если за 24 часа закрыть деплой, юридический контур и клиентские тексты, а за 72 часа — оплату (хотя бы Payment Link или счёт), вид кейса для Карен, канал связи и ручной red-flag протокол, первого клиента можно провести через платформу на этом ядре.

**Полный in-platform цикл** (оплата с webhook, сообщения, AI-сводка, автоматический red-flag workflow) за 7 дней недостижим — эти блоки отсутствуют полностью, а не «частично».
